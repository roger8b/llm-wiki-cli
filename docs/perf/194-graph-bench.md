# Graph view performance — issue #194

Benchmark captured against a synthetic brain with **1000 pages / ~3000 `[[wikilinks]]`**
(generated by `scripts/gen_synthetic_wiki.py`). Captured 2026-06-14 on the
`feat/194-graph-performance` branch.

## Numbers

| Stage                                | Time        | Notes                                 |
|--------------------------------------|-------------|---------------------------------------|
| `GET /api/graph` (cold)              | 476 ms      | First hit; SQLite + payload build     |
| `GET /api/graph` (warm, 5 runs avg)  | ~468 ms     | Stable; size 322 521 B                |
| `runLayout` cold, alpha = 1          | 407 ms      | 1000 nodes / 3000 links, full sim     |
| `runLayout` warm, alpha = 0.3        | 276 ms      | Reused cached positions               |
| `runLayout` with `onTick` callback   | 413 ms      | 29 throttled snapshots emitted        |

The layout runs in a dedicated Web Worker (`ui/src/workers/graphLayout.worker.ts`)
so the main thread stays responsive — only `requestAnimationFrame`-throttled
position snapshots reach React state. With 1000 nodes, the worker finishes in
under half a second on a developer laptop and the canvas rerender is
imperceptible.

The previous implementation ran a hand-rolled O(n²) force loop with up to 350
iterations on the main thread; for 1000 nodes that would land in the
multi-second range and freeze the UI. See PR #194 for the new code path.

## How to reproduce

```bash
# 1) Create a scratch brain and activate it.
BRAIN="/tmp/synthwiki-perf-$$"
mkdir -p "$BRAIN"
BRAIN_ID=$(curl -s -X POST http://localhost:8000/api/brains/create \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"synth-perf\",\"path\":\"$BRAIN\",\"activate\":true}" \
  | python -c "import sys,json;print(json.load(sys.stdin)['id'])")

# 2) Seed 1000 synthetic pages with ~3 links each.
.venv/bin/python scripts/gen_synthetic_wiki.py "$BRAIN" 1000 3

# 3) Time the API.
curl -s -o /dev/null -w "graph api: %{time_total}s\n" http://localhost:8000/api/graph

# 4) Time the layout in node.
( cd ui && npx tsx /tmp/bench_layout.mjs )

# 5) Open http://localhost:5173/graph in the browser; 1000 nodes will be
#    rendered on the <canvas> path (>300 nodes) — toggle types, search,
#    and click a node to enter focus mode.

# 6) Clean up.
curl -s -X POST http://localhost:8000/api/brains/active \
  -H "Content-Type: application/json" \
  -d '{"id":"<your-real-brain-id>"}' >/dev/null
rm -rf "$BRAIN"
```

## What the new code does

- `ui/src/workers/graphLayoutCore.ts` — pure `runLayout` using `d3-force`
  (`forceManyBody` / `forceLink` / `forceCenter`). Drives ticks itself so the
  worker (not d3's timer) owns the loop; emits throttled snapshots via
  `onTick`. Unit-tested (`graphLayoutCore.test.ts`).
- `ui/src/workers/graphLayout.worker.ts` — thin `postMessage` wrapper; receives
  a `LayoutRequest`, streams `{type:"tick"}` then `{type:"done"}`.
- `ui/src/views/GraphCanvas.tsx` — imperative `<canvas>` renderer past 300
  nodes. Hit-tests pointer events by nearest node within a radius; draws
  edges → nodes → labels (only the highest-degree handful + hovered /
  matched / selected node).
- `ui/src/lib/graphPositions.ts` — per-brain position cache in
  `localStorage`. Cached nodes start where they were, and a low simulation
  alpha keeps the layout stable across reopens.
- `ui/src/views/GraphView.tsx` — alternates SVG (`≤300` nodes) and canvas
  (`>300`) using `CANVAS_THRESHOLD`; seeds from the cache, runs the worker
  with `alpha = cachedCount/length > 0.6 ? 0.3 : 1`.
- `scripts/gen_synthetic_wiki.py` — developer fixture, not a product
  feature; seeded with `random.seed(42)` for reproducible layouts.
